# NIP-38333: AI Agent Builds

`draft` `optional`

This NIP defines kind `38333`, a parameterized replaceable event for publishing and sharing AI agent configuration builds.

## Motivation

AI agent platforms let users configure agents with models, personas, skills, integrations, automations, and memory structures. There's no standard way to share these configurations across platforms or between users. Kind 38333 provides a decentralized, censorship-resistant format for publishing and discovering agent builds.

## Event Format

Kind 38333 is a [NIP-33](33.md) parameterized replaceable event. The `d` tag serves as the build identifier. Publishing a new event with the same `d` tag replaces the previous version.

```json
{
  "kind": 38333,
  "content": "<build JSON>",
  "tags": [
    ["d", "<build-name>"],
    ["t", "<category-tag>"],
    ["client", "<app-name>", "<app-version>"],
    ["e", "<parent-event-id>", "", "fork"],
    ["p", "<original-author-pubkey>"]
  ]
}
```

### Content

The `content` field contains a JSON string with the build definition. The build format uses a block-based structure:

```json
{
  "schema": 2,
  "meta": {
    "name": "Build Name",
    "agentName": "Agent Name",
    "author": "<npub or display name>",
    "description": "What this agent does",
    "version": 1,
    "exportedAt": "2026-03-12T00:00:00Z"
  },
  "blocks": {
    "model": { "default": "claude-sonnet-4-5", "tiers": {} },
    "persona": { "name": "Agent", "role": "assistant" },
    "skills": { "items": [] },
    "integrations": { "items": [] },
    "automations": { "heartbeat": [], "cron": [] },
    "memory": { "structure": [] }
  },
  "dependencies": {
    "bins": [],
    "brew": [],
    "pip": [],
    "npm": [],
    "models": [],
    "platform": []
  }
}
```

### Tags

| Tag | Required | Description |
|-----|----------|-------------|
| `d` | Yes | Build name (NIP-33 identifier). Unique per author. |
| `t` | No | Category tags for discovery (e.g., "personal", "developer", "smart-home"). |
| `client` | Yes | Application name and version that created the event. |
| `e` | Fork only | Event reference to the parent build, with marker `"fork"`. |
| `p` | Fork only | Pubkey of the original author being forked. |

### Fork Provenance

When a build is derived from another, the event includes provenance:

```json
["e", "<parent-event-id>", "", "fork"]
["p", "<original-author-pubkey>"]
```

Multiple `e` tags with the `"fork"` marker indicate a build that merges ideas from multiple parents.

## Client Behavior

### Publishing

1. Export the agent configuration into the build JSON format
2. Scrub PII (emails, phone numbers, API keys, file paths, IP addresses) from content before publishing
3. Sign and publish as a kind 38333 event

### Discovering

Query relays with `{"kinds": [38333]}`. Filter by `t` tags for category browsing. Use NIP-50 text search where supported.

### Applying

1. Fetch the event and parse the build JSON from `content`
2. Validate the build against the schema
3. Run security checks (prompt injection, automation safety, network exfiltration patterns)
4. Check and resolve dependencies
5. Create a new agent workspace and apply the configuration

## Security Considerations

Builds can contain executable content (shell commands in automations, skill code, integration configs). Clients MUST:

- Scan for PII leaks before publishing
- Validate builds against the schema before applying
- Warn users about automation blocks that execute shell commands
- Check skills against known malware databases when available
- Never auto-apply builds without user confirmation
